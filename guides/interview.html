// FUNDAMENTALS

Language Fundamentals

These appear in almost every Go interview. Know them cold.

Interview tip For any "what's the difference between X and Y" question, give a one-line summary first, then a concrete code example. Interviewers reward clarity over exhaustive enumeration.
Q01 What is the difference between var x int and x := 0? EASY ▶

Both declare an integer variable with value 0. The difference is scope and style: var x int is a full declaration usable at package level or inside a function. x := 0 is the short declaration — only valid inside a function body. The short form is idiomatic Go for local variables. var is preferred at package scope or when you want to be explicit about the zero value without assigning.

Q02 How does Go handle inheritance? Does it have classes? EASY ▶

Go has no classes and no inheritance. Instead it uses composition via embedding and interfaces via structural typing. You embed one struct inside another to reuse fields and methods. A type satisfies an interface simply by implementing its method set — no explicit declaration needed. This avoids the fragile base class problem and keeps dependencies loose.

type Animal struct{ Name string } type Dog struct{ Animal; Breed string } // embedding d := Dog{Animal: Animal{Name:"Rex"}, Breed:"Husky"} fmt.Println(d.Name) // promoted field — "Rex"
Q03 What is the zero value in Go, and why does it matter? EASY ▶

Every type in Go has a zero value — the default when a variable is declared without assignment. int → 0, string → "", bool → false, pointers/slices/maps/channels/functions → nil. This eliminates undefined behaviour — Go variables are always in a valid, usable state. It also enables idiomatic patterns like sync.Mutex which is ready to use without initialisation.

Q04 What is a slice, and how is it different from an array? EASY ▶

An array in Go has a fixed size that is part of its type — [3]int and [4]int are different types. A slice is a descriptor pointing to an underlying array, with a length and capacity — it grows dynamically via append. In practice you almost never use raw arrays; slices are the standard. The key gotcha: slices sharing the same backing array share mutations — use copy to break this link.

// Slice header = (pointer, len, cap) s1 := []int{1,2,3} s2 := s1 // same backing array! s2[0] = 99 // mutates s1[0] too s3 := make([]int, len(s1)) copy(s3, s1) // independent copy
Q05 How does Go handle errors? Why no exceptions? EASY ▶

Go treats errors as plain return values of type error (a built-in interface with one method: Error() string). Functions return (result, error). The caller must handle the error explicitly — it cannot be silently ignored like an exception that unwinds invisibly. This makes error handling visible in the call graph, makes APIs honest about what can fail, and eliminates unexpected control-flow jumps. Use fmt.Errorf("context: %w", err) to wrap errors with context while keeping them unwrappable.

Q06 What is the difference between a pointer receiver and a value receiver? MEDIUM ▶

A value receiver func (u User) Name() receives a copy of the struct — mutations inside the method don't affect the original. A pointer receiver func (u *User) SetName() receives a pointer — mutations persist. Use pointer receivers when: (1) the method mutates the receiver, (2) the struct is large and copying is expensive, or (3) you need consistent interface implementation across a type. Rule: if any method uses a pointer receiver, all methods should.

// CONCURRENCY

Concurrency Questions

The topic most Go interviewers go deepest on. Know the runtime model, not just the syntax.

Q07 What is a goroutine? How is it different from a thread? EASY ▶

A goroutine is a lightweight, cooperatively-scheduled execution unit managed by the Go runtime — not the OS. OS threads typically start with 1–8MB stacks; goroutines start at 2KB and grow on demand. The Go scheduler (M:N scheduler) multiplexes thousands of goroutines onto a small pool of OS threads (GOMAXPROCS defaults to CPU count). Context-switching between goroutines is 10–100× cheaper than OS thread context switches. This is why you can have hundreds of thousands of goroutines in one process.

Q08 What is a data race, and how do you detect one? MEDIUM ▶

A data race occurs when two goroutines access the same memory location concurrently and at least one access is a write, with no synchronisation between them. Data races cause undefined behaviour — corrupted data, crashes, or subtle bugs that only appear under load. Go ships a built-in race detector: run go test -race ./... or go run -race main.go. It instruments memory accesses at runtime and reports races with full stack traces. Always run -race in CI. Fix races with channels, sync.Mutex, or sync/atomic.

Q09 What is the difference between a buffered and unbuffered channel? MEDIUM ▶

An unbuffered channel (make(chan T)) requires both sender and receiver to be ready simultaneously — it's a synchronisation point. The send blocks until a receiver is ready, and vice versa. A buffered channel (make(chan T, N)) holds up to N values without a receiver. Sends only block when the buffer is full. Use unbuffered for strict synchronisation, buffered for decoupling producer/consumer rates or as a semaphore. Misusing either causes deadlocks or goroutine leaks.

Q10 What is a goroutine leak, and how do you prevent it? HARD ▶

A goroutine leak occurs when a goroutine is started but never exits — it blocks forever waiting on a channel, network call, or lock that will never complete. Over time, leaks exhaust memory. Prevention: always give goroutines a way to exit via context.Context cancellation or a done channel. Use goleak in tests to detect leaks automatically. The /debug/pprof/goroutine endpoint reveals stuck goroutines in production.

// Leak: goroutine blocks forever if ctx never cancels go func() { select { case <-ctx.Done(): return // clean exit case v := <-work: process(v) } }()
Q11 Explain the Go memory model and the happens-before relationship. HARD ▶

The Go memory model defines the conditions under which reads are guaranteed to observe a write. A read r of variable x can observe a write w if: w happens-before r and no other write happens between w and r. Synchronisation events that create happens-before edges include: goroutine creation (go), channel operations (send happens-before receive on unbuffered; close happens-before receive returning zero), mutex unlock before lock, sync.Once.Do. Without a happens-before edge, the compiler and CPU can reorder operations — meaning you can't rely on a write in one goroutine being visible in another without explicit synchronisation.

// RUNTIME

Runtime & Internals

Senior-level questions that separate those who've read about Go from those who've run it in production.

Q12 How does Go's garbage collector work? HARD ▶

Go uses a tri-color mark-and-sweep garbage collector that runs concurrently with the program. Objects are white (unreachable), grey (reachable, children not scanned), or black (reachable, children scanned). The GC starts marking from roots (stacks, globals), turns objects grey then black. Unreachable (white) objects are swept. The STW (stop-the-world) pauses are very short — typically under 1ms since Go 1.14 — because most marking happens concurrently. The GC is triggered when heap size doubles. Use GOGC env var to tune aggressiveness (default 100 = GC when heap doubles).

Q13 What is escape analysis, and why does it matter? HARD ▶

Escape analysis is a compile-time analysis that determines whether a variable can live on the stack or must be allocated on the heap. Stack allocations are free — the frame is popped when the function returns. Heap allocations require GC. A variable "escapes" to the heap when: it's returned as a pointer, stored in an interface, captured by a closure that outlives the function, or sent on a channel. Run go build -gcflags='-m' ./... to see exactly what escapes and why. Reducing escapes directly reduces GC pressure.

Q14 What does defer do, and when does it execute? MEDIUM ▶

defer schedules a function call to execute when the surrounding function returns — whether that return is normal, via return statement, or due to a panic. Multiple defers run in LIFO order. Arguments to a deferred function are evaluated immediately at the defer statement, not at execution time. Common uses: closing files/connections, unlocking mutexes, recovering from panics. The overhead is small but non-zero — avoid in hot inner loops.

func readFile(path string) error { f, err := os.Open(path) if err != nil { return err } defer f.Close() // runs when readFile returns // ... use f }
Q15 What are Go generics, and when should you use them? MEDIUM ▶

Go 1.18 introduced generics via type parameters. They let you write functions and types that work across multiple types without code duplication or losing type safety. Use generics when: you're writing a reusable data structure (a typed stack, queue, or set), a utility function that operates on any slice or map, or a library that needs to work across numeric types. Don't use generics to avoid interfaces when interfaces are simpler — the Go team motto is "if in doubt, don't."

// Generic function — works on any ordered type func Min[T constraints.Ordered](a, b T) T { if a < b { return a } return b } Min(3, 5) // int Min("a", "b") // string
