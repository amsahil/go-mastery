Concurrency &
Goroutines

Go's concurrency model is its superpower. Learn goroutines, channels, mutexes, and the patterns that make Go concurrent programs safe, efficient, and elegant.

// FOUNDATION

The Big Three

Go's concurrency is built on three primitives. Master these and you can build anything concurrent.

01

Goroutines

Lightweight threads managed by the Go runtime — not the OS. You can run thousands of goroutines in a single process. They start with 2KB of stack and grow dynamically.

go func()
02

Channels

Typed conduits for communication between goroutines. Channels synchronize execution and pass data safely without locks. "Don't communicate by sharing memory; share memory by communicating."

make(chan T)
03

Select

Like a switch statement for channels. Lets a goroutine wait on multiple channel operations simultaneously, executing whichever is ready first. Essential for timeouts and fan-in patterns.

select { case ... }
// GOROUTINES

Launching Goroutines

The go keyword is all you need. Prefix any function call and it runs concurrently.

goroutine_basics.go 
package main

import (
    "fmt"
    "sync"
    "time"
)

func worker(id int, wg *sync.WaitGroup) {
    defer wg.Done() // Signal completion when this goroutine finishes
    fmt.Printf("Worker %d starting\n", id)
    time.Sleep(time.Second)
    fmt.Printf("Worker %d done\n", id)
}

func main() {
    var wg sync.WaitGroup

    for i := 1; i <= 5; i++ {
        wg.Add(1)
        go worker(i, &wg) // Launch 5 goroutines concurrently
    }

    wg.Wait() // Block until all goroutines finish
    fmt.Println("All workers done")
}
✓
Always use sync.WaitGroup for coordination Never use time.Sleep() to "wait" for goroutines. Use WaitGroup.Wait() to block the main goroutine until all workers have finished their work.
// CHANNELS

Channels & Communication

Channels are how goroutines talk to each other. Think of a channel as a typed pipe — one goroutine sends, another receives.

channels.go — producer / consumer pattern 
package main

import "fmt"

// Producer sends values into the channel
func producer(ch chan<- int) {
    for i := 0; i < 5; i++ {
        ch <- i // Send i into channel
    }
    close(ch) // Always close when done producing
}

// Consumer reads values from the channel
func consumer(ch <-chan int) {
    for v := range ch { // Range stops when channel is closed
        fmt.Printf("Received: %d\n", v)
    }
}

func main() {
    ch := make(chan int, 5) // Buffered channel — capacity 5
    go producer(ch)
    consumer(ch) // Runs in main goroutine
}
select.go — timeout pattern 
func fetchWithTimeout(url string) (string, error) {
    result := make(chan string, 1)

    go func() {
        // Simulate HTTP fetch
        result <- fetch(url)
    }()

    select {
    case res := <-result:
        return res, nil
    case <-time.After(3 * time.Second):
        return "", fmt.Errorf("request timed out")
    }
}
⚠
Buffered vs Unbuffered Channels An unbuffered channel make(chan T) blocks the sender until a receiver is ready. A buffered channel make(chan T, N) allows N sends before blocking. Choose carefully — wrong buffering leads to deadlocks or goroutine leaks.
// SYNC PACKAGE

Mutexes & Sync Primitives

When channels are overkill, the sync package provides classic concurrency tools — used correctly.

mutex_safe_counter.go 
type SafeCounter struct {
    mu    sync.Mutex
    count int
}

func (c *SafeCounter) Increment() {
    c.mu.Lock()         // Acquire lock
    defer c.mu.Unlock() // Always release — even on panic
    c.count++
}

func (c *SafeCounter) Value() int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.count
}

// For read-heavy workloads, use sync.RWMutex:
// c.mu.RLock() / c.mu.RUnlock() for reads
// c.mu.Lock()  / c.mu.Unlock() for writes

Patterns at a Glance

Common concurrency patterns, when to use them, and what to watch out for.

Pattern Use When Primitive Watch Out For
Worker Pool Limit parallel work, process job queue Goroutines + channel Channel full → deadlock
Fan-Out / Fan-In Parallelize multiple fetches, merge results Multiple goroutines + select Goroutine leaks
Pipeline Streaming data transformation stages Chained channels Back-pressure needed
Done Channel Cancellation propagation context.Context or chan struct{} Idiomatic Go approach
Mutex Guard Shared state with simple read/write sync.Mutex / RWMutex Lock contention
Once Lazy initialization, singletons sync.Once Safe for concurrent init
Context Deadlines, cancellation, request-scoped data context.Context Always pass as first arg
// CONTEXT PACKAGE

Context — Cancellation & Deadlines

The context package is how Go propagates cancellation, deadlines, and request-scoped values across goroutines and API calls.

context_cancel.go 
func doWork(ctx context.Context) error {
    for {
        select {
        case <-ctx.Done():
            // Context cancelled or deadline exceeded
            return ctx.Err()
        default:
            // Do actual work
            processNextItem()
        }
    }
}

func main() {
    // Cancel after 5 seconds
    ctx, cancel := context.WithTimeout(
        context.Background(),
        5*time.Second,
    )
    defer cancel() // Always defer cancel to free resources

    if err := doWork(ctx); err != nil {
        fmt.Println("Work stopped:", err)
    }
}
✓
Context is the first argument, always By convention, context.Context is always the first parameter of any function that accepts it — never stored in a struct. This makes cancellation chains explicit and traceable.
