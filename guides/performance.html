// PPROF

Profiling with pprof

Go ships a production-grade profiler in the standard library. Add five lines to expose it — then use the flame graphs to find exactly what's slow.

main.go — expose pprof endpoint
import (
    _ "net/http/pprof" // blank import registers pprof handlers
    "net/http"
)

func main() {
    // Debug server on separate port — never expose to public
    go func() {
        http.ListenAndServe("localhost:6060", nil)
    }()
    // ... rest of your app
}
bash — capture and analyze profiles
# CPU profile — 30 second sample
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# Heap memory profile
go tool pprof http://localhost:6060/debug/pprof/heap

# Goroutine trace
go tool pprof http://localhost:6060/debug/pprof/goroutine

# Open flame graph in browser
go tool pprof -http=:8081 cpu.prof

# Benchmark then profile
go test -bench=. -cpuprofile=cpu.prof ./...
go tool pprof cpu.prof

// MEMORY

Memory, Allocation & the GC

Go's garbage collector is concurrent and low-pause, but understanding how to reduce allocations is the single most impactful performance lever you have.

sync.Pool — reuse allocations
// Pool reduces pressure on GC by
// reusing objects across goroutines
var bufPool = sync.Pool{
    New: func() any {
        return make([]byte, 0, 4096)
    },
}

func processRequest() {
    buf := bufPool.Get().([]byte)
    buf = buf[:0] // reset length
    defer bufPool.Put(buf)

    // use buf safely...
}
pre-allocate slices
// BAD: grows repeatedly, many allocs
result := []int{}
for i := 0; i < 1000; i++ {
    result = append(result, i)
}

// GOOD: single allocation
result := make([]int, 0, 1000)
for i := 0; i < 1000; i++ {
    result = append(result, i)
}

// Check escape analysis:
// go build -gcflags='-m' ./...
⚠
Measure before optimizing Write benchmarks with testing.B first. Use go test -bench=. -benchmem to see allocations per operation. Only optimize the hot path that pprof actually identifies — premature optimization is real.
// GOLDEN RULES

10 Rules for Fast Go Code

Lessons from production Go at scale — from Cloudflare, Google, Uber, and the community.

01

Profile first, optimize second

Use pprof to find the actual bottleneck. Guessing wastes time and often makes code worse. Measure, then act.

02

Pre-allocate slices when size is known

make([]T, 0, n) avoids repeated doubling and copying. This is often a 2–5× win for large collections.

03

Reuse buffers with sync.Pool

For hot paths that allocate temporary buffers (JSON encoding, HTTP bodies), Pool eliminates GC pressure.

04

Avoid small interfaces on hot paths

Interface method calls prevent inlining. For tight loops, consider concrete types or function pointers.

05

Prefer []byte over string for I/O

Converting string↔[]byte allocates. When reading/writing network or file data, stay in []byte throughout.

06

Use strings.Builder for concatenation

String + string allocates on every operation. strings.Builder grows a single buffer.

07

Keep goroutines coarse-grained

Spawning thousands of goroutines per request increases scheduler overhead. Use worker pools to bound concurrency.

08

Avoid fmt.Sprintf in hot paths

fmt.Sprintf uses reflection and allocates. Use strconv for number conversions and strings.Builder for assembly.

09

GOMAXPROCS defaults to CPU count

Go uses all available CPUs by default. In containers, set it explicitly with runtime.GOMAXPROCS or the automaxprocs library.

10

Use escape analysis to reduce heap allocations

Run go build -gcflags='-m' to see what escapes to heap. Stack allocations are free — heap allocations cost GC cycles.

// BENCHMARKING

Writing Benchmarks

Go benchmarks are built into the test runner. Write them alongside your tests — they're the only way to know if your optimizations actually help.

string_bench_test.go
func BenchmarkConcatPlus(b *testing.B) {
    for i := 0; i < b.N; i++ {
        s := ""
        for j := 0; j < 100; j++ {
            s += "x" // allocates on each iteration
        }
        _ = s
    }
}

func BenchmarkConcatBuilder(b *testing.B) {
    for i := 0; i < b.N; i++ {
        var sb strings.Builder
        for j := 0; j < 100; j++ {
            sb.WriteString("x")
        }
        _ = sb.String()
    }
}

// go test -bench=BenchmarkConcat -benchmem
// BenchmarkConcatPlus-8     500000    2840 ns/op   5040 B/op   99 allocs/op
// BenchmarkConcatBuilder-8  3000000    420 ns/op     96 B/op    2 allocs/op
✓
Always use -benchmem The -benchmem flag shows allocations per operation alongside timing. Reducing allocations is often more impactful than reducing CPU time, because allocations directly drive GC pressure.
